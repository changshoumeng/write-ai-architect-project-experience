# 输入契约

本文件用于判断输入是否足以支撑 `write-ai-architect-project-experience` 的两个工作流。

## 工作流输入要求

| 工作流 | 必需输入 | 可选但建议 |
|---|---|---|
| Generate Case | 目标 JD、项目描述/项目材料 | 输出文件、目标公司/行业、可公开边界、证据材料 |
| Review And Rewrite Case | 已有项目经历文档 | 目标 JD、原始项目材料、输出文件、目标岗位方向 |

缺少必需输入时，必须先追问，不要直接生成或重写最终稿。

## 项目材料可以包括

- 项目经历详细稿、项目复盘、PRD、架构文档。
- 代码路径、测试记录、评测结果、交付记录。
- 性能、成本、缓存、配额、Trace、日志、用户行为埋点、使用频次和反馈样本记录。
- 原业务流程、AI 改造后流程、用户入口、任务流、结果交付/回写/验收链路。
- 用户口述的项目背景、职责、技术方案、交付状态。
- 可写入简历的业务结果、指标、客户反馈或上线范围。

## JD 信息可以包括

- 岗位原文。
- 招聘链接摘录或截图转文字。
- 用户整理的岗位职责、任职要求、关键词、技术栈和加分项。

## 必须抽取的项目事实

- 项目名称、项目时间、项目角色。
- 目标用户、业务场景、核心痛点、项目目标。
- 原业务流程、AI 改造后流程、用户入口到结果交付的任务流。
- 个人职责：产品定义、方案设计、研发交付。
- 核心技术：技术栈、Agent架构设计、AI 工程化。
- AI 工程化：垂直场景 System Prompt、Prompt/Schema、工具契约、评测样本、验证修正、重试、回退、Trace、Guardrails、业务系统集成、日志观测、性能/成本优化、行为埋点和价值评估。
- 工程交付：核心模块、接口联调、测试、文档、PoC/MVP/上线/验收边界。
- 高风险表述：数字、生产状态、客户名称、团队规模、ROI、独立主导。

## JD 能力抽取

从 JD 中抽取并排序：

- **Skills**：技术栈、工具、框架、工程方法，如 LangGraph、RAG、FastAPI、Vue、Docker、CI/CD。
- **Knowledge**：AI Agent、LLM、RAG、知识图谱、模型评测、系统架构、业务领域知识。
- **Abilities**：业务抽象、架构设计、复杂问题拆解、技术选型、跨团队协作、排障、交付。
- **Other Characteristics**：owner 意识、沟通影响力、风险意识、学习能力、客户导向、结果导向。

## 项目事实到 JD 的映射

对每个关键能力项标记：

| 标记 | 含义 | 写法 |
|---|---|---|
| 强匹配 | 项目事实能直接证明该能力 | 放入核心技术或职责成果 |
| 弱匹配 | 项目材料提到，但证据不足 | 保守表达，不写硬指标 |
| 缺口 | JD 重要，但项目材料无事实支撑 | 追问用户补充；无补充则不写 |
| 不可写 | 无证据、涉密或明显夸大 | 删除或匿名化 |

## 证据强度分级

| 级别 | 证据 | 可用表述 |
|---|---|---|
| L3 强证据 | 代码、架构文档、测试、评测、交付记录直接支撑 | `设计并落地`、`实现`、`验证`、`交付` |
| L2 叙述证据 | 项目材料明确写到，但缺少独立佐证 | 可写 `负责/推动`，高风险内容需保守 |
| L1 弱证据 | 只能从上下文推断 | `形成方案`、`完成 PoC`、`参与规划`，或不写 |
| L0 无证据 | 没有来源 | 不写 |

## 必须追问的情况

- Generate Case 没有 JD 或项目描述。
- Review And Rewrite Case 没有已有项目经历文档。
- 项目角色或职责边界不清。
- JD 最关键能力在项目材料里没有对应事实。
- 用户想写量化指标，但未提供指标来源。
- 用户想写客户名称、模型规格、数据量、上线范围，但公开边界不清。

## 保守表达库

- `完成 PoC/原型/MVP 验证`
- `打通关键业务链路`
- `形成可复用方案`
- `沉淀 Agent 工作流或工程实践`
- `支撑后续上线/推广/迭代`
- `建立评测与观测基础`
- `在材料范围内可证明`
- `仍需补充验收记录/评测报告/代码路径进一步佐证`
