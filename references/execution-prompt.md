# 执行口径

本文件定义 `write-ai-architect-project-experience` 的写作与审查方法。该 skill 只服务两个工作流：

1. 根据 JD + 项目描述生成符合模板的项目经历。
2. 对已有项目经历做百分制审查，并重写为符合模板的项目经历。

模板文件只定义最终输出结构；本文件定义如何分析、审查和写作。

## 角色视角

写作和审查时同时模拟以下视角，但不要在最终项目经历正文里逐项暴露这些角色：

- AI 技术架构师：系统边界、架构取舍、AI Workflow、工程复杂度。
- AI Agent 工程专家：Agent 职责、状态、工具、Prompt/Schema、Trace、重试、回退、Guardrails。
- 简历策略顾问：HR 首屏可读性、岗位命中率、表达密度。
- 乙方交付负责人：范围控制、证据边界、交付状态、风险表达。
- 技术面试官：内容是否一目了然、是否经得起追问。

## Workflow A: Generate Case

输入：目标 JD + 项目描述/项目材料。

执行步骤：

1. 抽取 JD 关键信息：
   - Skills：技术栈、工具、框架、工程方法。
   - Knowledge：AI Agent、RAG、LLM、系统架构、业务领域、质量工程。
   - Abilities：业务抽象、架构设计、复杂问题拆解、排障、交付。
   - Other Characteristics：owner 意识、风险意识、沟通影响力、可信度。
2. 抽取项目事实：
   - 项目名称、项目时间、项目角色。
   - 目标用户、业务场景、原业务流程、AI 改造后流程、核心痛点、项目目标。
   - 技术栈、Agent架构设计、用户入口到结果交付的任务流、Agent/RAG/LLM 工作流、工具和外部系统。
   - AI 工程化：垂直场景 System Prompt、Schema、工具契约、验证/修正、性能/成本优化、Trace、评测、重试、回退、Guardrails、业务系统集成、日志观测、行为埋点和价值评估。
   - 个人职责边界、研发交付、验证状态和证据来源。
3. 建立 `JD 能力要求 -> 项目事实 -> 可写段落` 映射：
   - 强匹配：放入核心技术或职责成果。
   - 弱匹配：保守表达。
   - 缺口：追问用户；无补充则不强写。
   - 不可写：删除或匿名化。
4. 使用 `assets/project-experience-case-template.md` 组织正文。
5. 运行校验并按质量门禁重写薄弱段落。

## Workflow B: Review And Rewrite Case

输入：已有项目经历文档；JD 和项目材料可选，但若用户提供，应纳入审查。

输出必须包含：

1. 百分制评分。
2. 主要问题清单。
3. 修改建议。
4. 重写后的项目经历正文，且正文符合 `assets/project-experience-case-template.md`。

审查步骤：

1. 先按模板结构检查：项目介绍、核心技术、关键职责与成果是否完整。
2. 再按 HR/面试官视角判断：是否能快速看懂项目是什么、原流程如何被 AI 改造、为什么重要、我做了什么、技术含金量在哪里。
3. 若提供 JD，检查 JD 关键词和能力要求是否被项目事实自然覆盖。
4. 检查 AI 架构深度：Agent架构设计与 AI 工程化是否具体，尤其是垂直场景调优、验证闭环、性能/成本优化和观测评估是否有事实。
5. 检查可信度：职责边界、交付状态、指标、客户认可、上线范围是否有证据。
6. 给出具体问题和修改方向，再重写最终版本。

问题清单应具体，例如：

- `核心技术过于罗列技术名词，没有解释系统分层和工具边界。`
- `职责成果只写“完成开发”，缺少研发交付、评测或验证闭环。`
- `JD 要求 Agent 工程化能力，但正文没有体现垂直场景 Prompt/工具契约、验证修正、性能优化或 Trace/埋点评估。`

避免泛泛建议，例如“语言再优化”“内容再丰富”。

## 项目经历正文写法

使用 `assets/project-experience-case-template.md`。建议总长度约 700-1100 字，可按用户要求调整。

- `项目介绍`：120-200 字，2-3 句话讲清背景、目标用户、痛点、建设内容和价值。
- `核心技术`：350-550 字，覆盖技术栈、Agent架构设计、AI 工程化。
- `关键职责与成果`：250-400 字，覆盖产品定义、方案设计、研发交付。

`研发交付` 合并开发落地、联调、评测、文档、验收和迭代支撑。它不是简单写“完成开发”，而是写从方案到可运行、可验证、可迭代成果的闭环。

`AI 工程化` 不要只写 Prompt 和 Trace。优先从项目事实中挑 2-4 类高含金量机制：垂直业务场景的 System Prompt/工具契约、结构化或视觉验证、Reflection/重试/回退、缓存/流式/上下文压缩/token 控制等性能成本优化、Trace/日志/行为埋点/使用频次/反馈样本等观测与价值评估。

## 高质量样稿的可复用模式

生成或重写前，先在内部确认正文能回答 6 个问题；最终正文不必逐项列出这些问题，但必须让读者读完自然得到答案：

1. 原业务流程是什么，为什么低效或不可控？
2. AI 改造后，用户从哪里进入，经过什么任务流，最终得到什么可编辑、可复用或可验证的结果？
3. 系统为什么采用当前架构，而不是更重或更轻的替代方案？
4. Agent 如何被 Prompt/Schema/工具契约约束，如何避免生成不可验证的结果？
5. 工程化上如何处理验证修正、性能成本、Trace 日志、行为数据或价值评估？
6. 作者本人负责哪些设计、实现、联调、验证或交付物，哪些只是团队或平台能力？

写作时把这些答案压缩成自然段落，避免把所有候选机制都塞进正文。没有证据的机制宁可不写。

## 写作风格

目标风格是自然、专业、连贯、克制，像真实人类专家写给招聘方和面试官看的项目经历。

应该做到：

- 先讲业务场景和问题，再讲技术方案。
- 用因果关系连接事实，例如“因为存在 X 约束，所以选择 Y 方案，并通过 Z 控制风险”。
- 技术术语必须解释其作用，不做名词堆叠。
- 写清个人职责边界，区分 `主导`、`负责`、`参与`、`协同`。
- 用可追问的细节增强可信度，如模块边界、工具契约、Trace、评测样本、异常处理和交付状态。

避免：

- 机械排比、模板腔、宣传腔、每句都套“通过 A，基于 B，实现 C”。
- 空泛动词堆叠，如连续使用“赋能、打造、沉淀、闭环、提升、优化”但没有事实。
- 无证据强指标、客户认可、生产上线、规模化推广。
- 强写不存在的多 Agent、MCP、RAG、评测平台或复杂架构。

## 证据表达

- 强证据：可写 `设计并落地`、`实现`、`验证`、`交付`。
- 中等证据：可写 `负责`、`参与设计`、`推动落地`，避免硬指标。
- 弱证据：写 `形成方案`、`完成 PoC/MVP`、`打通关键链路`、`支撑后续迭代`。
- 无证据或涉密：不写，或匿名化后保守表达。
