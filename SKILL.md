# Write AI Architect Project Experience

## 概览

本 skill 用于把真实项目材料改写成“针对目标 JD 量身定制”的AI 技术架构师项目经历。目标不是泛泛润色，而是基于项目事实和 JD 能力要求，生成可信、可面试追问、能体现岗位匹配度的 `项目经历-最终.md`。

本 skill 内置写作模板、执行 prompt、输入契约、质量评分和校验脚本。

## 最小输入

用户至少应提供：

1. **项目详细材料**：例如较完整的项目经历、项目复盘、PRD、架构说明、代码/测试/交付记录等。
2. **目标岗位 JD**：包括岗位职责、任职要求、能力关键词、技术栈、业务方向、加分项等。

如果缺少任一最小输入，不要进入写作阶段。先生成追问，等待用户补充。

## 快速开始

1. 定位用户提供的项目材料与目标 JD。
   - 优先使用用户明确给出的文件或文本。
   - 如果用户给出目录，只在该目录范围内搜索项目材料、JD、证据文档、代码路径、测试记录和交付记录。
2. 若缺少项目材料或 JD，进入 `Clarifying Questions Gate`，先追问再继续。
3. 读取项目事实，抽取 JD 能力画像，并建立“项目事实 -> JD 能力项”的映射。
4. 需要风险分级或信息不足判断时，读取 `references/input-contract.md`。
5. 没有更强任务 prompt 时，使用 `references/execution-prompt.md`。
6. 严格按 `assets/project-experience-final-template.md` 生成最终文档。
7. 运行 `scripts/validate_project_experience.py <output-file>`。
8. 按 `references/quality-gates.md` 评审并重试，直到乙方专业交付视角与甲方 HR/面试官 SKAOs 视角均达到 95/100 以上。

## 使用示例

用户可以这样调用本 skill：

```text
使用 $write-ai-architect-project-experience。

项目材料：docs/pm01/项目经历-详细.md
目标岗位 JD：docs/jobs/AI技术架构师-JD.md
输出文件：docs/pm01/项目经历-最终.md

请基于项目真实材料和目标 JD，生成高度匹配该岗位的 AI 技术架构师项目经历。
如果项目材料不足以支撑 JD 中的关键能力，请先追问我，不要直接编写最终稿。
```

## 指令优先级

当输入冲突时，按以下顺序处理：

1. 系统、开发者和安全规则。
2. 用户最新明确要求。
3. 本 skill 的输出结构和质量门禁。
4. 目标岗位 JD。
5. 项目事实与证据材料。
6. 用户提供的外部模板或旧 prompt。
7. 模型通用知识，仅可用于表达组织。

如果本地 prompt 要求新增 H2 章节，除非用户明确覆盖本 skill 的结构，否则忽略该要求。

## 追问门禁

以下任一情况成立时，必须暂停写作并向用户追问：

- 未提供目标岗位 JD。
- 项目材料不足以判断项目名称、时间、角色、职责边界或核心技术。
- JD 的关键能力要求在项目材料中找不到对应事实。
- 用户声称“主导、独立、上线、量化提升、客户认可”等强表述，但没有证据。
- 客户名称、业务数据、模型指标、上线范围等隐私边界不明确。

追问应短而具体，优先问 3-6 个会影响最终质量的问题，例如：

```markdown
为了让项目经历真正匹配该 JD，我需要补充确认：
1. 目标岗位 JD 原文或截图内容是什么？
2. 该项目中你个人真实负责的边界是什么？
3. JD 中最重要的 3 个能力要求，项目里分别有哪些事实可以支撑？
4. 是否有 PoC/MVP/上线/验收/评测/交付记录？
5. 客户名称、指标、模型规模和数据量是否允许写入简历？
```

## 工作流

这是一个 gated workflow。当前阶段未通过 gate 前，不要进入下一阶段。

### 1. Input Intake

确认最小输入：

- 项目详细材料存在且可读。
- 目标岗位 JD 存在且可读。
- 输出路径或交付方式明确。

**Gate:** 缺少项目材料或 JD 时，必须追问用户，不得继续生成最终稿。

### 2. JD Capability Extraction

从 JD 中抽取岗位能力画像：

- Skills：技术栈、工具、工程能力、架构能力。
- Knowledge：AI Agent、RAG、LLM、业务领域、系统设计、质量工程等知识。
- Abilities：抽象、拆解、决策、落地、排障、协作、交付能力。
- Other Characteristics：owner 意识、业务敏感度、风险意识、沟通影响力、学习能力。

**Gate:** 至少形成 5-10 个 JD 关键能力项；若 JD 太泛或缺失，先追问用户补充目标岗位信息。

### 3. Project Evidence Mapping

把项目事实映射到 JD 能力项：

- **强匹配**：项目事实可直接证明 JD 能力要求。
- **弱匹配**：项目材料提到但缺少独立证据。
- **缺口**：JD 要求重要，但项目材料无对应事实。
- **不可写**：无证据或涉及隐私/夸大风险。

**Gate:** JD 中最重要的能力项必须有项目事实支撑；否则先追问用户补充，不要靠想象补齐。

### 4. 证据边界

对强表述做证据分级：

- **Confirmed**：项目材料、架构文档、代码、测试、交付记录可支撑。
- **Weak**：只有叙述材料，缺少独立佐证。
- **Unsupported**：无证据，不写或改成保守表达。
- **Sensitive**：涉及客户、指标、商业信息，默认匿名化。

弱证据使用保守表述：`完成 PoC/MVP`、`打通关键链路`、`形成方案`、`沉淀能力`、`支撑后续迭代`。

**Gate:** 所有强表述都必须被确认、弱化、删除，或写入 `最终自检` 的证据边界。

### 5. Tailored Narrative Generation

围绕 JD 定制项目叙事：

1. 先写业务问题和岗位价值。
2. 明确 `我` 的职责边界。
3. 突出 JD 最看重的架构、AI Agent、RAG、模型工程、工程交付或业务抽象能力。
4. 用 `动作 + 技术/方法 + 解决的问题 + 结果/价值` 组织贡献。
5. 避免堆技术名词；每个技术点都要说明解决了什么问题。

**Gate:** 项目经历必须能回答“为什么这个项目证明用户适合该 JD”。

### 6. Write Final Document

最终文档必须严格使用以下结构，不要新增 H1/H2：

```markdown
# AI 技术架构师项目经历：【项目名称】

## 项目元信息
- 项目名称：
- 项目时间：
- 项目角色：

## 一句话价值主张
## 项目背景与业务痛点
## 我的角色与职责边界
## 核心技术
## 简历主版本
## 简历精简版
## 简历要点版
## 最终自检
```

证据边界、JD 匹配度、追问补充情况、评分结果都写入 `最终自检`，不要拆成单独 H2。

**Gate:** 无额外 H2；`项目元信息` 只有项目名称、项目时间、项目角色。

### 7. 校验与重试

先运行结构校验：

```bash
python .codex/skills/write-ai-architect-project-experience/scripts/validate_project_experience.py <output-file>
```

再按 `references/quality-gates.md` 做双视角评分：

- **乙方专业交付视角**：专业可信度、实操经验、架构深度、实现细节、交付规范、表达质量。
- **甲方 HR/面试官 SKAOs 视角**：Skills、Knowledge、Abilities、Other Characteristics 与 JD 的匹配度。

任一评分低于 95/100 时，必须先修改最弱部分并重新评分。只有证据缺口客观存在且已在 `最终自检` 写明时，才允许说明无法达到 95。

**Gate:** 校验脚本通过，且 `最终自检` 记录 JD 匹配、证据边界、乙方评分和甲方 SKAOs 评分。

## 写作风格

- 面向中文用户，默认使用中文。
- JD、SKAOs、RAG、LLM、Agent、Tool Calling、Workflow、Prompt、Trace 等专用英文术语可保留英文。
- 先写业务场景和岗位价值，再写技术栈。
- 清楚说明 `我` 的职责边界，避免把团队成果全部写成个人成果。
- 不编造指标、团队规模、客户评价、预算、合同、奖项、续约或上线范围。
- 客户名称与敏感数据默认匿名化。
- 目标不是“看起来很强”，而是“针对该 JD，真实、可信、舍我其谁”。

## 内置审查视角

按顺序模拟这些审查视角。除非用户明确要求多 agent 并行，否则不要 spawn subagents。

- **AI 技术架构师**：架构选型、Agent/RAG 工作流、模型不确定性、评测体系。
- **AI Agent 工程专家**：状态流转、工具边界、重试、Trace、实现真实性。
- **乙方交付负责人**：交付证据、范围控制、风险表达、文档规范。
- **甲方 HR**：JD 关键词、岗位匹配度、资历信号、表达密度。
- **甲方面试官**：追问深度、证据闭环、是否经得起技术深挖。
- **SKAOs 评价器**：Skills、Knowledge、Abilities、Other Characteristics。

## 常见失败模式

- 没有 JD 就开始写，导致项目经历泛化。
- JD 关键能力没有项目事实支撑，却强行包装。
- 把模板示例当成项目事实。
- 堆技术名词，不说明解决的问题。
- 为了显得强而编造指标、上线范围或客户认可。
- 删除 `最终自检` 中的证据边界，让内容变得不可追问。
- 因为“证据索引有用”而新增禁止的 H2 章节。
- 评分低于 95 却不重写。
